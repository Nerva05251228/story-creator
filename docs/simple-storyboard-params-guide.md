# 简单分镜 8 个参数说明

本文档说明当前 `manage` 界面中，对外可配置的 8 个“程序分镜规则”参数分别控制什么、会影响什么，以及实际调参时应该如何理解。

## 1. 先理解两个概念

### 单元

单元是程序先切出来的最小文本块，不是最终给用户看的镜头。

例如原文：

```text
悟空抬头看向山门。八戒拎着钉耙跟上来。沙僧回头确认师父还在。
```

可能先被切成 3 个单元：

```text
单元1：悟空抬头看向山门。
单元2：八戒拎着钉耙跟上来。
单元3：沙僧回头确认师父还在。
```

### 镜头

镜头是程序把多个单元按规则重新合并后的最终结果，也就是“简单分镜”里看到的内容。

例如上面的 3 个单元，最终可能合并成：

```text
镜头1：悟空抬头看向山门。八戒拎着钉耙跟上来。
镜头2：沙僧回头确认师父还在。
```

所以可以简单理解为：

- 单元：内部切分的最小块
- 镜头：最终输出的简单分镜

## 补充：单元和镜头的实际划分规则

这一节解释当前程序在代码里是怎么工作的，不是概念上的“理想规则”。

### 单元是怎么切出来的

程序会从左到右扫描整段正文，优先按照“强边界”切单元。

实际顺序大致是：

1. 先检查当前字符是不是成对符号的一部分  
   例如引号、括号、书名号、方括号。
2. 如果在成对符号内部，程序会尽量不切  
   也就是不会优先在引号或括号中间断开。
3. 程序会把换行或空行当作天然段落边界  
   只要当前不在成对符号内部，遇到这里就可以切出一个单元。
4. 如果遇到强边界标点，就优先在这里切  
   而且会优先匹配更长的标点组合。
5. 如果第一轮切出来的单元太长，再做二次拆分  
   先尝试继续按强边界拆；如果还偏长，再退一步按弱边界拆。

这里有两个实现细节很重要：

- 强边界按“长 token 优先”匹配  
  所以 `？！` 会被当成一个整体，而不是先切 `？` 再切 `！`。
- 切点后面的闭合符号会尽量跟前一个单元走  
  例如 `。”`、`？”` 这类组合，会尽量整体留在前一个单元里。

### 镜头是怎么由单元合并出来的

单元切出来后，程序不会立刻把每个单元当成一个镜头，而是继续往后合并。

合并时主要看 4 组条件：

1. 当前镜头是否还没达到目标字数下限  
   如果还没达到，程序通常会继续往后并。
2. 再并一个单元会不会超过目标字数上限  
   如果已经达到下限，并且再并会超过目标上限，程序会倾向停下。
3. 再并一个单元会不会超过软上限  
   超过后会明显倾向切开。
4. 再并一个单元会不会超过硬上限  
   超过就必须切开。

除此之外，还有两个额外限制：

- 每镜头最多单元数  
  单元数到上限时，即使字数还没超，也会停下。
- 短碎片合并阈值  
  如果最终某个镜头太短，会尽量并回前一个镜头，避免碎成很短的小镜头。

### 一个完整例子

原文：

```text
她沉默了……“你真的要走？！”他追问。她没回头...
```

第一步，切单元时，程序通常会得到：

```text
单元1：她沉默了……
单元2：“你真的要走？！”
单元3：他追问。
单元4：她没回头...
```

原因是：

- `……` 是强边界
- `？！` 是一个整体强边界
- `。` 是强边界
- `...` 也是强边界
- 引号内容不会优先从中间拆开

第二步，再按镜头规则合并，假设当前参数偏短、每镜头最多 2 个单元，那么可能得到：

```text
镜头1：她沉默了……“你真的要走？！”
镜头2：他追问。她没回头...
```

### 再看一个长句例子

原文：

```text
她低声提醒我先别慌，再把当年的事情一件件说清楚，又叮嘱我先把账册都重新查一遍。
```

如果整句太长，程序会先尝试按强边界切。  
但因为这句内部没有句号、问号、叹号这类强边界，所以第一轮可能还是一整块。

这时如果已经超过软上限，程序会再按弱边界去拆，结果可能变成：

```text
单元1：她低声提醒我先别慌，
单元2：再把当年的事情一件件说清楚，
单元3：又叮嘱我先把账册都重新查一遍。
```

然后再根据字数和单元数规则，把这些单元重新并成镜头。

### 一句话总结

可以把当前实现理解成：

- 先用“边界规则”切出尽量自然的单元
- 再用“字数和数量规则”把单元拼成镜头

所以这 8 个参数并不是直接决定“每一镜切哪里”，而是：

- 一部分参数决定“哪里适合切成单元”
- 另一部分参数决定“几个单元适合并成一镜”

## 2. 参数分组

这 8 个参数可以分成两组：

### A. 控制镜头长度和合并节奏

- 目标字数下限
- 目标字数上限
- 软上限
- 硬上限
- 每镜头最多单元数
- 短碎片合并阈值

### B. 控制程序优先在哪些地方切

- 强边界标点
- 弱边界标点

## 3. 八个参数逐项说明

### 3.1 目标字数下限

含义：

- 一个镜头希望至少有这么多字。
- 如果当前镜头还没到这个值，程序通常会继续把后面的单元并进来。

例子：

原文：

```text
悟空抬头看向山门。八戒拎着钉耙跟上来。沙僧回头确认师父还在。
```

如果 `目标字数下限 = 35`，程序通常不会很早停下，可能得到：

```text
镜头1：悟空抬头看向山门。八戒拎着钉耙跟上来。
镜头2：沙僧回头确认师父还在。
```

如果把它改得很小，比如 `20`，程序更容易提前结束当前镜头，结果可能变成：

```text
镜头1：悟空抬头看向山门。
镜头2：八戒拎着钉耙跟上来。
镜头3：沙僧回头确认师父还在。
```

调参直觉：

- 调大：镜头更长
- 调小：镜头更碎

### 3.2 目标字数上限

含义：

- 单个镜头最理想的上边界。
- 程序到了这个区间后，会更倾向于在当前合适边界停下来。

例子：

原文：

```text
他停下脚步。她没有回头。风从巷口灌进来。灯笼轻轻晃动。
```

如果 `目标字数上限 = 60`，程序可能会把前两句或前三句合并在一起。

如果改成 `30`，程序会更早结束镜头，结果可能是：

```text
镜头1：他停下脚步。她没有回头。
镜头2：风从巷口灌进来。灯笼轻轻晃动。
```

调参直觉：

- 调大：允许单镜头装更多内容
- 调小：镜头更容易被切开

### 3.3 软上限

含义：

- 超过这个值后，程序会明显倾向于切分。
- 但这不是绝对强制，如果上下文边界不好，程序仍会做有限让步。

例子：

原文：

```text
嫡母揉了揉额角，似乎有些头疼于我的愚钝，但语气却放缓下来，她低声提醒我先别慌，再把当年的事情一件件说清楚，又叮嘱我先把宫里的眼线和账册都重新查一遍。
```

如果 `软上限 = 70`，程序会尽量在逗号等弱边界处分开，例如：

```text
镜头1：嫡母揉了揉额角，似乎有些头疼于我的愚钝，但语气却放缓下来，
镜头2：她低声提醒我先别慌，再把当年的事情一件件说清楚，
镜头3：又叮嘱我先把宫里的眼线和账册都重新查一遍。
```

调参直觉：

- 调大：长句更容易留在同一镜头里
- 调小：程序会更积极地把长句拆开

### 3.4 硬上限

含义：

- 单个镜头的绝对安全上限。
- 只要再往里并会明显超过这个值，程序就会强制拆开。

例子：

原文：

```text
悟空：师父，前面妖气太重，我们不能再往前走了，石阶尽头埋伏的人已经按住了兵器，只等我们踏进山门就会一起冲出来。
```

如果这句明显超过 `硬上限 = 90`，程序会强制拆成两段，而不会为了“看起来是一整句对白”而无限保留：

```text
镜头1：悟空：师父，前面妖气太重，我们不能再往前走了，
镜头2：石阶尽头埋伏的人已经按住了兵器，只等我们踏进山门就会一起冲出来。
```

调参直觉：

- 调大：允许特别长的镜头存在
- 调小：会更严格地拆长句

### 3.5 每镜头最多单元数

含义：

- 一个镜头最多能由几个单元拼起来。
- 即使字数还没超，如果单元数达到上限，也会停止继续合并。

例子：

原文：

```text
他停下脚步。她没有回头。风从巷口灌进来。灯笼轻轻晃动。
```

如果切出来是 4 个单元，并且 `每镜头最多单元数 = 2`，那么最多会是：

```text
镜头1：他停下脚步。她没有回头。
镜头2：风从巷口灌进来。灯笼轻轻晃动。
```

即使总字数还不算很大，也不会 4 个单元全塞进一个镜头里。

调参直觉：

- 调大：镜头更容易吞下更多句子
- 调小：镜头节奏更快，结构更清晰

### 3.6 短碎片合并阈值

含义：

- 如果某个镜头太短，短到低于这个值，程序会尽量把它并到前一个镜头里。
- 目的是避免出现“只剩几个字单独成镜”的情况。

例子：

原文：

```text
她转身就走。门开了。
```

如果后一句太短，而 `短碎片合并阈值 = 18`，程序更可能输出：

```text
镜头1：她转身就走。门开了。
```

而不是：

```text
镜头1：她转身就走。
镜头2：门开了。
```

调参直觉：

- 调大：程序更不愿意保留很短的独立镜头
- 调小：允许更多短促、碎片化镜头存在

### 3.7 强边界标点

含义：

- 这些标点被视为“最优先的切分点”。
- 程序在扫描文本时，首先会在这些地方尝试切出单元。

默认示例：

```text
。,！,？,；,……,...,？！,！？,!!,??
```

例子：

原文：

```text
她沉默了……“你真的要走？！”他追问。她没回头...
```

如果这些标点都在强边界里，程序通常会优先切成：

```text
单元1：她沉默了……
单元2：“你真的要走？！”
单元3：他追问。
单元4：她没回头...
```

调参直觉：

- 强边界越多：程序越容易在这些停顿点切开
- 强边界越少：程序更可能把长句继续往后带

### 3.8 弱边界标点

含义：

- 这些标点不是首选切点，但当句子太长、又找不到更合适的强边界时，程序会退而求其次在这里切。

默认示例：

```text
，,、,：,:
```

例子：

原文：

```text
她低声提醒我先别慌，再把当年的事情一件件说清楚，又叮嘱我先把账册都重新查一遍。
```

如果这一整句已经太长，程序可能按逗号拆成：

```text
单元1：她低声提醒我先别慌，
单元2：再把当年的事情一件件说清楚，
单元3：又叮嘱我先把账册都重新查一遍。
```

调参直觉：

- 弱边界越多：长句越容易被细拆
- 弱边界越少：程序更倾向保留完整句

## 4. 一组完整例子

### 示例原文

```text
悟空抬头看向山门。八戒拎着钉耙跟上来。沙僧回头确认师父还在。山风穿过石阶，旗幡猎猎作响。悟空压低声音说前面不对劲。八戒嘟囔着问要不要先撤。
```

### 情况 A：当前默认 15s 风格

参数特征：

- 目标字数下限 `35`
- 目标字数上限 `60`
- 软上限 `70`
- 硬上限 `90`
- 每镜头最多单元数 `2`

可能结果：

```text
镜头1：悟空抬头看向山门。八戒拎着钉耙跟上来。
镜头2：沙僧回头确认师父还在。山风穿过石阶，旗幡猎猎作响。
镜头3：悟空压低声音说前面不对劲。八戒嘟囔着问要不要先撤。
```

特点：

- 节奏偏快
- 单镜头内容较短
- 更适合短视频、短剧快节奏镜头

### 情况 B：把镜头调长

如果改成：

- 目标字数下限 `55`
- 目标字数上限 `90`
- 每镜头最多单元数 `3`

可能结果：

```text
镜头1：悟空抬头看向山门。八戒拎着钉耙跟上来。沙僧回头确认师父还在。
镜头2：山风穿过石阶，旗幡猎猎作响。悟空压低声音说前面不对劲。八戒嘟囔着问要不要先撤。
```

特点：

- 单镜头更完整
- 节奏更稳
- 更适合叙事平缓的文案

### 情况 C：把长句拆得更碎

如果把 `软上限` 改小，且保留逗号作为弱边界：

- 软上限 `50`
- 弱边界保留 `，`

那么类似：

```text
山风穿过石阶，旗幡猎猎作响。
```

这种长句在必要时就更容易被拆开。

## 5. 调参建议

### 想让简单分镜更碎、更快

可以优先：

- 降低 `目标字数上限`
- 降低 `软上限`
- 降低 `每镜头最多单元数`

### 想让简单分镜更稳、更整

可以优先：

- 提高 `目标字数下限`
- 提高 `目标字数上限`
- 提高 `每镜头最多单元数`

### 想避免超长句拖成一整镜

可以优先：

- 降低 `软上限`
- 保留或增加弱边界标点

### 想避免出现特别短的小碎镜

可以优先：

- 提高 `短碎片合并阈值`

## 6. 当前默认值的整体风格

当前 15s 规则默认值：

```text
目标字数下限：35
目标字数上限：60
软上限：70
硬上限：90
每镜头最多单元数：2
短碎片合并阈值：18
```

整体效果可以概括成：

- 优先按完整句子切
- 长句必要时允许在逗号处拆
- 单镜头偏短
- 节奏偏快
- 同时尽量避免只剩几个字的碎镜头

这套默认值更适合“短剧式、节奏较快、单镜头不拖沓”的简单分镜输出。
